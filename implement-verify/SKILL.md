# Implement Verify

## 目标

根据已确认的业务设计和详细设计实施编码，并通过验证动作给出编码报告。

## 状态规则

- 当前状态根据动作显式输出：`编码实现中` 或 `验证中`
- 本阶段默认主动推进，但高风险实现分歧必须及时拉用户确认
- 当实现过程中出现方案偏移、风险修复策略、兼容性处理分歧时，必须改用选择题形式向用户确认

## 执行规则

- 编码前先简述本轮目标与范围
- 编码后必须做验证，不允许“写完就算完成”
- 若实现结果与设计明显不符，必须回退并说明原因
- 若存在两种以上实现路径且会影响质量、成本或风险，先给推荐项，再让用户选择

## 输出契约

必须在 `编码报告` 目录输出编码报告，至少包含：

1. 当前实现目标
2. 涉及文件/模块
3. 关键实现点
4. 与原设计的偏差
5. 验证动作
6. 验证结果
7. 已知问题与剩余风险
8. 是否可进入 review-hard
9. 如发生实现分歧，附“实现选择题”

## 回退规则

- 设计缺失 -> 回退 `code-detailed-design`
- 业务规则不成立 -> 回退 `business-design`
- 现实约束与理解冲突 -> 回退 `repo-investigation`

## 禁止事项

- 禁止跳过验证
- 禁止以主观感觉代替验证结果
- 禁止未说明偏差就私自改设计
