# Review Hard

## 目标

在开发完成候选阶段，对实现结果做严格审查，优先找 bug、回归风险、设计偏差和缺失测试。

## 状态规则

- 当前状态固定输出为：`严格审查中`
- 审查结果必须以问题为中心，而不是先写总结
- 当审查后存在多种修复路径、回退路径或“先修哪个”的分歧时，必须以选择题形式交给用户确认

## 审查重点

- 功能正确性
- 回归风险
- 边界条件
- 权限与安全问题
- 并发与一致性风险
- 日志与可观测性缺口
- 测试缺口
- 与业务设计/详细设计的偏差

## 输出契约

必须输出：

1. 审查结论
2. 按严重级别排序的问题清单
3. 需要回退的阶段
4. 如存在多条修复路径，给出“修复选择题”
5. 可以进入 `completion-workflow` 的条件
6. 若无问题，也必须说明残余风险或验证缺口

## 回退规则

- 发现实现问题 -> 回退 `implement-verify`
- 发现设计问题 -> 回退 `code-detailed-design` 或 `business-design`
- 发现项目理解偏差 -> 回退 `repo-investigation`

## 禁止事项

- 禁止只做表扬式总结
- 禁止发现问题但不指向应回退阶段
- 禁止因为“已经改完了”就降低审查标准
