# kaifa-skill

> 一个以“提高开发者参与度”为核心的协作式开发 Skill 系统

`kaifa-skill` 不是一个“让 LLM 直接写代码”的提示词集合，而是一套围绕真实开发流程组织的 Skill 体系。\
它的核心目标不是替开发者做决定，而是让 LLM 在开发过程中持续提出方案、展示状态、给出选项，并把关键决策交还给开发者。

## 为什么做这个项目

很多 AI Coding Prompt 的问题不在于“代码写不出来”，而在于：

- 一上来就跳过理解和设计，直接开始编码
- 过程黑箱，开发者不知道当前处于哪个阶段
- 关键业务和技术决策由 LLM 单方面拍板
- 做完之后没有设计沉淀、没有报告、没有历史回顾
- 复杂需求一旦中途偏航，很难回退和纠偏

`kaifa-skill` 想解决的正是这些问题。

## 核心理念

这个项目只做三件事：

- 先让 LLM **显式说明当前状态**
- 再让 LLM **给出推荐和备选方案**
- 最后由开发者 **在关键节点做选择**

一句话概括：

> 不是“把开发交给 LLM”，而是“让 LLM 在结构化流程里协助开发者决策和执行”。

## 核心卖点

### 1. 提高开发者参与度

这是整个项目最重要的设计目标。

所有涉及业务规则、技术选型、流程分支、风险处理、回退路径、归档命名的地方，都要求 LLM 采用“选择题”形式输出，而不是只给一个结论。

标准形式如下：

```markdown
当前状态：...
当前目标：...
我的推荐：方案 A
推荐理由：...

请你做选择：
1. 方案 A（推荐） - 适用条件 / 收益 / 风险
2. 方案 B - 适用条件 / 收益 / 风险
3. 方案 C - 适用条件 / 收益 / 风险

如果你不选，我的默认动作：...
你确认后，我下一步会：...
```

这意味着：

- LLM 负责整理方案和比较取舍
- 开发者负责关键判断和拍板
- 协作过程可见、可控、可回退

### 2. 强制状态显式化

每个 Skill 都必须明确说明当前状态，例如：

- 路由判断中
- 项目调查中
- 业务设计中
- 详细设计中
- 编码实现中
- 验证中
- 严格审查中
- 收尾归档中
- 回退修正中

这样做的价值很直接：

- 开发者随时知道当前走到哪一步
- 复杂任务不会在阶段之间无声跳转
- 对话过程更容易复盘和归档

### 3. 阶段化开发，而不是直接生成代码

`kaifa-skill` 默认不鼓励“直接开写”。

它把开发流程拆成明确阶段：

```text
dev-routing
-> repo-investigation
-> business-design
-> code-detailed-design
-> implement-verify
-> review-hard
-> completion-workflow
```

只有在开发者明确同意时，才允许跳过前置阶段直接编码。

### 4. 设计、实现、审查、归档一体化

项目不仅关注“写出代码”，还要求沉淀完整过程资产，包括：

- 项目理解报告
- 业务设计
- 编码报告
- 历史回顾
- 设计失败原因

这样做的意义是：

- 不是一次性问答
- 不是写完即结束
- 而是让每轮开发都能留下可复用的上下文和决策记录

## 核心流程

### 新项目

适用于从 0 开始的功能或模块设计。

```text
business-design
-> code-detailed-design
-> implement-verify
-> review-hard
-> completion-workflow
```

### 已有代码项目

适用于已有仓库、已有模块、已有业务背景的场景。

```text
repo-investigation
-> business-design
-> code-detailed-design
-> implement-verify
-> review-hard
-> completion-workflow
```

之所以已有代码项目必须优先进入 `repo-investigation`，是因为：

- 设计必须建立在当前真实实现之上
- 不能只根据用户描述或抽象理解做方案
- 一旦代码或业务范围更新，项目理解也需要刷新

## 7 个核心 Skill

| Skill                  | 作用                   |
| :--------------------- | :------------------- |
| `dev-routing`          | 判断当前任务应走哪条流程，并显式声明状态 |
| `repo-investigation`   | 生成项目理解报告，刷新对项目现状的认知  |
| `business-design`      | 做需求理解、业务闭环和验收口径设计    |
| `code-detailed-design` | 把业务设计转成模块、接口、数据和实现方案 |
| `implement-verify`     | 执行编码并输出编码报告          |
| `review-hard`          | 严格审查实现质量，优先发现风险和缺口   |
| `completion-workflow`  | 做收尾检查，沉淀历史回顾文档       |

## 与普通 AI Coding Prompt 的区别

| 普通 Prompt | kaifa-skill     |
| :-------- | :-------------- |
| 直接产出代码    | 先路由、再调查、再设计、再实现 |
| LLM 主导决策  | 开发者参与关键决策       |
| 状态不透明     | 每一步显式标明当前状态     |
| 很少沉淀过程    | 强制输出报告和历史回顾     |
| 更适合简单问答   | 更适合复杂开发与多人协作思路  |

## 输出目录

项目当前设计了 5 个核心输出目录：

| 目录       | 用途                    |
| :------- | :-------------------- |
| `设计失败原因` | 记录方案不自洽、现实约束冲突、设计回退原因 |
| `历史回顾`   | 存放开发完成后的流程总结          |
| `业务设计`   | 存放业务设计文档及其配套详细设计      |
| `项目理解报告` | 存放对项目现状的理解和调查结果       |
| `编码报告`   | 存放编码过程、验证动作和结果        |

这些目录的意义不是“多写文档”，而是让开发过程真正具备：

- 可追溯性
- 可复盘性
- 可复用性

## 适用场景

`kaifa-skill` 更适合以下任务：

- 中大型功能开发
- 已有项目的二次开发
- 复杂需求分析
- 高风险改动
- 需要业务设计和代码设计分层推进的任务
- 希望让 AI 融入规范开发流程的个人或团队

## 不适用场景

以下场景通常不值得使用完整流程：

- 一次性简单脚本
- 小范围、低风险的机械修改
- 纯查询类问题
- 不需要设计、验证和归档的快速问答

## 项目结构

```text
kaifa-skill/
├── .trae/skills/
│   ├── dev-routing/
│   ├── repo-investigation/
│   ├── business-design/
│   ├── code-detailed-design/
│   ├── implement-verify/
│   ├── review-hard/
│   └── completion-workflow/
├── templates/
├── 设计失败原因/
├── 历史回顾/
├── 业务设计/
├── 项目理解报告/
└── 编码报告/
```

- `.trae/skills/`：Skill 主体
- `templates/`：文档模板
- 各输出目录：用于沉淀开发过程资产

## 方法论总结

`kaifa-skill` 的核心方法很简单：

- 先理解，再设计，再编码
- 先推荐，再选择，再推进
- 先验证，再审查，再归档

它不追求“让 LLM 替开发者做完一切”，而是追求：

> 让 LLM 成为一个透明、可控、会给方案、会等待确认、会沉淀结果的协作开发助手。

## 结尾

如果你不希望 AI 一上来就直接改代码，而是希望它先讲清楚自己在做什么、为什么这么做、有哪些可选路径、风险分别是什么，并把关键决策交给你，那么 `kaifa-skill` 适合你。
